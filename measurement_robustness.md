## Discussion of the partial data

The headline finding from this partial run is **null** at the resolution
the experiment can support. Across the 12 conditions completed, every
final val accuracy lies within $\pm 0.011$ of the 0.269 unperturbed
baseline. That spread is plausibly within the within-seed run-to-run
noise at this scale (6000 train images, 5 epochs, single seed) — so we
**cannot** claim from this data that any of the tested bias values
($|\delta_{\rm sub}| \le 0.10$ for 1c, $\delta_{\rm leak} \le 0.10$ for
1a) damages learning.

**This contradicts the prior expectation** (set in the experiment plan)
that 1c subtraction-bias would be the most sensitive variant. At least
on this partial data, 1c degrades no faster than 1a, and both stay
within the noise floor over the swept range.

There are two non-mutually-exclusive interpretations:

1. The rule really is robust to a few percent of measurement bias on
   these knobs, and the spec'd sweep range is too narrow to find a
   break point. If true, the circuit-co-design guidance shifts from
   "tighten 1c bias spec" to "any of these are fine to several percent;
   look elsewhere for the dominant non-ideality."
2. The single-seed × 6k-train × 5-epoch baseline is under-resolved (the
   noise floor is ~0.02 in val acc), so a real $\sim 1$%-of-baseline
   degradation is buried under it. The 60k-train Section-4 baseline
   (0.522 val acc, ~5× higher distance from chance) would resolve much
   smaller effects.

The overnight rerun should cover both: complete the missing six sweeps
at the same scale to see whether **any** of them produces a clear
break, and if 1c/1a still look flat at scale-1 then re-run the
"interesting" points at full Section-4 scale (60k) with 3 seeds before
drawing any silicon-design conclusions.

## How to finish the run

```bash
# Resume the full sweep (about 5 hours, single CPU):
python -u experiments/measurement_robustness/run_sweep.py \
    --sweep all --epochs 5 --train-size 6000

# Regenerate plots + tables + this report:
python -u experiments/measurement_robustness/analyze.py
```

The CSV writer appends to existing `results/<sweep>.csv`, so resuming
will duplicate the already-completed 1c rows; either delete those CSVs
first or accept the duplicates and dedupe in `analyze.py`.
